# Design: Change 019 - Just-In-Time Secrets Vault

## Summary
Secrets stay in host memory and are only exposed to component guests through a typed WIT import. Access is granted per sealed route via `allowed_secrets` stored inside `integrity.lock`, which is generated by `tachyon-cli`.

## Decisions

### Sealed authorization model
- Extend each sealed route with an optional `allowed_secrets` array.
- Keep the existing `user` and `system` execution roles unchanged.
- Restrict secret grants to user routes so telemetry/system routes keep their current execution model.

### Runtime model
- `core-host` loads a mock in-memory vault when compiled with `--features secrets-vault`.
- `faas-guest` components always import `secrets-vault`, and the host always satisfies that import.
- Without the feature, the host returns `vault-disabled` instead of failing component instantiation.

### Isolation model
- Component WASI contexts are created without inheriting host environment variables.
- `guest-example` checks both code paths:
  - `std::env::var("DB_PASS")` stays unset
  - `secrets-vault.get-secret("DB_PASS")` returns either the secret, `permission-denied`, or `vault-disabled`

### Authoring workflow
- `tachyon-cli generate` gains `--secret-route /path=NAME[,NAME]`.
- The checked-in `integrity.lock` is regenerated through `tachyon-cli` so the new route grant remains signed.
