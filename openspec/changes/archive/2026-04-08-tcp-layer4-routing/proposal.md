# Proposal: Change 053 - Layer 4 TCP Routing & Stream Pass-Through

## Context
Until now, Tachyon Mesh has been an HTTP/HTTPS (Layer 7) router. However, to host services like a custom Git forge (SSH - Port 22) or databases (PostgreSQL - Port 5432), the PaaS must handle raw TCP connections. Because WebAssembly (WASI Preview 1) lacks native asynchronous server socket support, the Rust Host must act as a Layer 4 Proxy, bridging raw TCP streams to WASM instances.

## Objective
1. Allow the `integrity.lock` configuration to map raw TCP ports to specific FaaS targets.
2. Implement the "WASM inetd pattern": The Rust host accepts the TCP connection and pipes the bidirectional byte stream directly into the WASM instance's standard I/O (or a specific WASI stream descriptor).
3. Manage the lifecycle: The FaaS instance lives exactly as long as the TCP connection is active, preserving the Scale-to-Zero philosophy for long-lived connections.

## Scope
- Update the configuration parser to accept `tcp_ports` bindings.
- Implement a `tokio::net::TcpListener` pool in the `core-host`.
- Wire the TCP `ReadHalf` to WASI `stdin` and the TCP `WriteHalf` to WASI `stdout`.

## Success Metrics
- A client can run `ssh -p 2222 git@tachyon-node`. The Rust host accepts the connection, spawns `system-faas-ssh.wasm`, and the user sees a custom SSH banner generated by the WASM code.
- When the client types `exit`, the TCP connection drops, and the WASM instance is immediately destroyed by the Garbage Collector.
