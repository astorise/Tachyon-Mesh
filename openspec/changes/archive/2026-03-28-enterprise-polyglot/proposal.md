# Proposal: Change 012 - Enterprise Polyglot FaaS (Java & C#)

## Why
While Rust, Go, and JavaScript prove the polyglot nature of Tachyon Mesh, the enterprise market relies heavily on Java and C# (.NET). To achieve true enterprise readiness, our FaaS host must be able to load and execute `.wasm` modules compiled from these managed languages without requiring external JVMs or CLRs on the host.

## What Changes
Introduce two new guest FaaS functions in the workspace: one written in C# (published as a .NET 8 `wasi-wasm` single-file module) and one written in Java (compiled via TeaVM to `WEBASSEMBLY_WASI`). Deploy and test these alongside the other FaaS guests in our CI integration pipeline.

## Scope
- Create a `guest-csharp` directory targeting .NET 8 `wasi-wasm` and publish it as `guest_csharp.wasm`.
- Create a `guest-java` directory using Maven and the TeaVM WASI target to emit `guest_java.wasm`.
- Lock in the existing generic host route-to-module mapping for `/api/guest-csharp` and `/api/guest-java` with focused tests instead of adding per-language branches.
- Expand the Dockerfile and `integration.yml` GitHub Action to install the managed-language toolchains, compile the modules, and assert their HTTP responses.
- Raise the sealed guest fuel budget so managed-language guests can boot while remaining bounded by quota enforcement.

## Success Metrics
- `curl http://localhost:8080/api/guest-csharp` returns a response generated by the C# code.
- `curl http://localhost:8080/api/guest-java` returns a response generated by the Java code.
- The `core-host` successfully executes these heavier garbage-collected `.wasm` modules without memory leaks or crashes.
